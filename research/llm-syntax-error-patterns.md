## 2. 오류 분류 체계 (Error Taxonomy)

벤치마크에서 관찰된 24건의 실패를 분석하여 6개의 대분류를 도출했다.

### 2.1 분류 체계

```
LLM 코드 생성 오류
├── A. 구문 오류 (Syntax Errors)
│   ├── A1. 잘못된 토큰/연산자 사용
│   ├── A2. 누락된 구분자/종료자
│   └── A3. 초기화 누락
├── B. 패턴 누출 (Pattern Leakage)
│   ├── B1. 타 언어 키워드/구문 유입
│   ├── B2. 타 언어 관용구(idiom) 유입
│   └── B3. 타 언어 선언 패턴 유입
├── C. 절단 오류 (Truncation Errors)
│   ├── C1. 출력 토큰 한도 미달 절단
│   └── C2. 생성 조기 종료
├── D. 형식 오염 (Format Contamination)
│   ├── D1. 마크다운 코드 펜스 혼입
│   └── D2. 비코드 텍스트 혼입
├── E. 의미 오류 (Semantic Errors)
│   ├── E1. 미정의 변수/함수 사용
│   └── E2. 타입 불일치
└── F. 컨텍스트 윈도우 오류 (Context Window Errors)
    ├── F1. 프롬프트 제약 조건 망각
    └── F2. 프롬프트 오류 정보 그대로 수용
```

### 2.2 벤치마크에서 관찰된 오류 분포

| 오류 유형 | 건수 | 비율 | 해당 LLM |
|-----------|------|------|----------|
| **A1. 잘못된 연산자** (`+=`) | 6 | 25% | GPT-4o(2), Gemini(4) |
| **A3. 초기화 누락** (`let x: Type`) | 2 | 8.3% | GPT-4o(2) |
| **B1. 타 언어 키워드** (`while`) | 2 | 8.3% | GPT-4o(2) |
| **C1. 출력 절단** | 3 | 12.5% | Gemini(3) |
| **C2. 조기 종료** (unterminated string) | 1 | 4.2% | Gemini(1) |
| **D1. 마크다운 혼입** | 2 | 8.3% | Gemini(2) |
| **E1. 토큰 오류** (unexpected NEWLINE) | 1 | 4.2% | Gemini(1) |
| **Python py_compile 오류** (혼합) | 7 | 29.2% | Gemini(7) |

## 3. 오류 유형별 심층 분석

### 3.1 A1: 잘못된 연산자 사용 (`+=` 문제)

**발생**: GPT-4o 2건, Gemini 4건 (총 6건, 전체의 25%)

**원인 구조**:
- 시스템 프롬프트(vibe-context.ts)에 `Assignment: = += -= *= /=`라고 명시
- 실제 Vibe 파서에는 `+=` 미구현
- LLM은 프롬프트 지시를 정확하게 따랐으나, 프롬프트 자체가 오류

**실제 코드 사례** (GPT-4o, twin_stick_dodge.vibe):
```
-- GPT-4o가 생성한 코드 (실패)
bx[i] = bx[i] + bvx[i] * dt   -- 이것은 통과
survived_time += dt             -- 이것은 실패 (파서 미지원)
```

**대응하는 Claude 코드** (성공):
```
-- Claude가 생성한 코드 (성공)
survived_time = survived_time + dt   -- 명시적 재할당
```

**핵심 인사이트**:
1. LLM은 프롬프트에 명시된 문법을 적극 활용한다 -- 프롬프트 오류는 곧 생성 오류
2. `+=`는 Python, JavaScript, C, Java, Go 등 거의 모든 주류 언어에 존재 -- 학습 데이터 압력이 강함
3. 프롬프트에서 제거하더라도 LLM이 다른 언어 패턴에서 `+=`를 가져올 가능성 있음

**언어 설계 함의**:
- **Option A**: `+=`를 실제로 구현 (학습 데이터 압력에 순응)
- **Option B**: `x = x + n` 만 허용하되, 프롬프트에 "no compound assignment" 명시
- **권장**: Option A. `+=`는 사실상 모든 언어의 보편적 패턴이므로, 이를 거부하면 LLM이 반복적으로 실패할 확률이 높음

### 3.2 A3/B3: 미초기화 변수 선언

**발생**: GPT-4o 2건

**오류 코드** (GPT-4o, twin_stick_dodge.vibe):
```
let spawn_x: Float          -- 파서 에러: expected EQ, got NEWLINE
let spawn_y: Float          -- 초기화 없는 변수 선언
```

**오류 코드** (GPT-4o, multi_wave_spawner.vibe):
```
let ex_pos: Float           -- 같은 패턴
let ey_pos: Float
```

**이것은 어디서 온 패턴인가?**

| 언어 | 미초기화 변수 선언 |
|------|-------------------|
| TypeScript | `let x: number;` -- 합법 |
| Python | 불가 (타입 힌트만: `x: float`) |
| Lua | `local x` -- 합법 (nil로 초기화) |
| Go | `var x float64` -- 합법 (zero value) |
| Rust | `let x: f64;` -- 합법 (사용 전 초기화 필수) |
| C/C++ | `float x;` -- 합법 |

**핵심 발견**: 미초기화 변수 선언은 **거의 모든 프로그래밍 언어에서 합법**이다. LLM에게 "Vibe에서는 반드시 초기화해야 한다"를 주입하기 매우 어려운 규칙이다.

**실제 패턴 분석**:
GPT-4o가 미초기화 변수를 사용한 맥락은 항상 동일했다:
```
let spawn_x: Float           -- 먼저 선언
let spawn_y: Float

if edge == 0                  -- 조건부 초기화
    spawn_x = rand_float(...)
    spawn_y = 0.0
else if edge == 1
    spawn_x = rand_float(...)
    spawn_y = 600.0
```

이것은 "조건부 초기화" 패턴이다. 변수를 먼저 선언하고, 여러 분기에서 각각 값을 할당한다. TypeScript, Go, C에서 매우 흔한 패턴이다.

**Claude는 어떻게 해결했는가?**
```
let sx: Float = 0.0           -- 기본값으로 초기화 후 덮어쓰기
let sy: Float = 0.0

if edge < 1.0
    sx = rand_float(...)
    sy = 0.0
...
```

Claude는 `= 0.0`으로 기본값을 주고 조건부로 덮어쓰는 방식을 택했다. 이는 프롬프트의 예제 코드에서 `let x: Float = 400.0` 패턴만 보여줬기 때문이다.

**언어 설계 함의**:
- **Option A**: 미초기화 변수 선언 허용 (타 언어와 호환, 패턴 누출 방지)
- **Option B**: 기본 zero 값 자동 초기화 (`let x: Float` = `let x: Float = 0.0`)
- **권장**: Option B. 미초기화를 허용하되 내부적으로 zero value를 할당. Go의 접근법과 동일. LLM은 타입만 알면 적절한 zero value를 추론할 수 있다.

### 3.3 B1: 타 언어 키워드 유입 (`while`)

**발생**: GPT-4o 2건 (twin_stick_dodge, multi_wave_spawner에서 관찰되나 파서 에러는 `+=`/초기화에서 먼저 발생)

**오류 코드** (GPT-4o, twin_stick_dodge.vibe):
```
while i < len(bx)             -- Vibe에는 while 없음 (for condition으로 대체)
    bx[i] = bx[i] + bvx[i] * dt
    ...
    i = i + 1
```

**Vibe 정상 코드** (Claude):
```
for i in range(0, len(bx))   -- for-in-range 루프 사용
    if blife[i] > 0.0
        bx[i] = bx[i] + bvx[i] * dt
```

**패턴 누출 분석**:
`while`은 Python, JavaScript, C, Go, Lua, Rust 등 **거의 모든 언어에 존재**하는 키워드다. Vibe가 `while`을 `for condition`으로 통합한 설계 결정은 혁신적이지만, LLM의 학습 데이터 압력과 직접 충돌한다.

| 언어 | while 문법 |
|------|-----------|
| Python | `while x > 0:` |
| JavaScript | `while (x > 0) { }` |
| Lua | `while x > 0 do ... end` |
| Go | `for x > 0 { }` (Go도 while 없이 for로 통합) |

Go가 흥미로운 선례다: Go도 `while`을 제거하고 `for`로 통합했으나, Go는 LLM 학습 데이터에 충분히 있어서 LLM이 이 규칙을 자연스럽게 따른다. Vibe는 학습 데이터가 0이므로 프롬프트만으로 이 규칙을 강제해야 한다.

**프롬프트에서의 문제**: vibe-context.ts의 for 루프 설명을 보면:
```
for x > 0                 -- condition loop (replaces while)
    x = x - 1
```
"replaces while"이라는 주석이 있으나, GPT-4o는 이를 무시하고 `while`을 사용했다.

**언어 설계 함의**:
- `while`을 완전히 제거한 설계는 **프롬프트로 강제하기 어려운** 규칙에 해당
- 그러나 Lexer 단계에서 `while`을 발견하면 `for`로 자동 변환하거나 명확한 에러 메시지를 제공할 수 있음
- **권장**: `while`을 예약어로 등록하되 파싱 시 `for` 문과 동일하게 처리 (호환 레이어). 또는 별칭으로 허용.

### 3.4 C1: 출력 절단 (Truncation)

**발생**: Gemini 3건 (circle_collision, orbital_mechanics, chain_reaction)

**절단된 코드 사례**:

```
-- circle_collision (Gemini, 24 tokens, 파일 끝)

```
리스트 선언의 닫는 `]`가 없이 코드가 끝남.

```
-- orbital_mechanics (Gemini, 20 tokens, 파일 끝)
let orb_speeds: List[Float] = [1.5, 1.0
```
같은 패턴. 리스트 리터럴 중간에서 절단.

```
-- chain_reaction (Gemini, 37 tokens)
cx = cx + [rand_
```
함수 호출 중간에서 절단.

**토큰 수 분석**:

| 태스크 | 난이도 | Gemini 토큰 | GPT-4o 토큰 | Claude 토큰(추정) |
|--------|--------|------------|------------|-----------------|
| circle_collision | medium | **24** | 132 | ~130 |
| orbital_mechanics | hard | **20** | 112 | ~120 |
| chain_reaction | hard | **37** | 237 | ~240 |

Gemini의 절단된 출력은 GPT-4o/Claude 대비 **5~10배 적은 토큰**을 생성했다. 이는 API 타임아웃이나 토큰 한도가 아닌, **Gemini 자체의 생성 조기 종료** 문제다.

**왜 Gemini는 조기 종료하는가?**

벤치마크 설정: `maxOutputTokens: 2048`, `temperature: 0.2`

2048 토큰 한도는 충분히 높으므로, 문제는 Gemini의 내부 생성 메커니즘에 있다:
1. **Gemini 2.0 Flash는 긴 코드 생성에 약함**: 20~37 토큰에서 조기 종료
2. **익숙하지 않은 문법에서 "포기"**: 학습 데이터에 없는 Vibe 문법을 생성하다 EOS 토큰 확률이 급격히 상승
3. **비교**: 같은 태스크의 Lua-LOVE 버전에서 Gemini는 절단 없이 완전한 코드를 생성 (학습 데이터 존재)

**핵심 발견**: **출력 절단은 LLM이 "자신감이 없을 때" 발생하는 현상이다.** 학습 데이터에 없는 문법을 생성해야 할 때, 모델이 자신 없는 다음 토큰 대신 종료를 선택한다.

**Gemini 성공 vs 실패 토큰 수 비교**:
| 결과 | 평균 토큰 | 범위 |
|------|----------|------|
| 성공 (10건) | 55.2 | 25~131 |
| 실패-절단 (3건) | **27.0** | 20~37 |
| 실패-기타 (7건) | 32.6 | 21~38 |

성공한 경우에도 평균 55 토큰으로, GPT-4o(161)나 Claude(추정 ~170)에 비해 매우 적다. Gemini는 Vibe 코드를 극도로 압축적으로 생성하거나, 또는 불완전하게 생성한다.

### 3.5 D1: 마크다운 코드 펜스 오염

**발생**: Gemini 2건 (state_machine_game, platformer_level)

**오류 코드** (Gemini, state_machine_game.vibe):
```
```vibe                       <-- 마크다운 백틱이 코드에 포함됨
let state: Int = 0
let score: Int = 0
...
```

**오류 코드** (Gemini, platformer_level.vibe):
```
```                           <-- 언어 태그 없는 백틱
let px: Float = 100.0
...
```

**왜 이런 오류가 발생하는가?**

LLM의 학습 데이터에서 코드는 대부분 마크다운 컨텍스트에서 표현된다 (GitHub README, Stack Overflow, 블로그 등). 따라서 LLM은 "코드를 출력하라" = "마크다운 코드 블록으로 감싸라"로 해석하는 강한 경향이 있다.

**프롬프트 방어**:
현재 프롬프트: `Output ONLY the Vibe code, no explanations or markdown`

이 지시에도 불구하고 Gemini는 마크다운을 포함했다. **GPT-4o와 Claude는 이 지시를 정확하게 따랐다.**

**벤치마크의 방어 메커니즘**:
`llm-client.ts`의 `stripCodeFences()` 함수가 마크다운 펜스를 제거하지만, Gemini의 경우 코드가 백틱으로 시작하되 닫는 백틱 없이 절단되어 정규식 매칭에 실패했다.

**핵심 발견**: 마크다운 오염은 **모델의 instruction-following 능력**과 직결된다. GPT-4o(0건), Claude(0건), Gemini(2건) -- Gemini의 instruction following이 가장 약함.

### 3.7 Python-Pygame + Gemini 실패 (12건)

Python은 LLM 학습 데이터에서 가장 풍부한 언어임에도, Gemini는 40%만 성공했다. 실패 원인을 분류하면:

## 4. 패턴 누출(Pattern Leakage) 심층 분석

### 4.1 언어별 누출 강도 순위

LLM이 코드를 생성할 때, 다른 언어의 패턴이 침투하는 빈도와 강도를 측정했다.

#### 가설 검증: "학습 데이터가 많은 언어일수록 누출 강도가 강하다"

| 소스 언어 | 학습 데이터 규모(추정) | 누출 강도 | 대표적 누출 패턴 |
|-----------|---------------------|-----------|----------------|
| **Python** | 최대 (수억 파일) | **최강** | 들여쓰기, `def`, `self`, `:`, 미초기화 변수 |
| **JavaScript/TypeScript** | 매우 큼 | **강** | `const`/`let`, `===`, `null`, `undefined`, `+=` |
| **C/C++/Java** | 큼 | **중** | `{}`, `;`, 포인터 구문, `for(init;cond;step)` |
| **Lua** | 중간 | **약** | `local`, `end`, `then`, 1-기반 인덱싱 |
| **Go** | 중간 | **약** | `:=`, `func`, `if err != nil` |

#### Vibe에서 관찰된 누출

| 누출 패턴 | 소스 언어 | 발생 LLM | 발생 건수 |
|-----------|-----------|---------|-----------|
| `+=` 복합 할당 | Python/JS/C/모두 | GPT-4o, Gemini | 6 |
| `let x: Type` (초기화 없음) | TypeScript/Rust/Go | GPT-4o | 2 |
| `while` 키워드 | Python/JS/C/Lua/모두 | GPT-4o | 2 |
| 마크다운 백틱 | 학습 데이터 형식 | Gemini | 2 |

### 4.2 Python의 지배적 영향

Python은 LLM 학습 데이터에서 가장 큰 비중을 차지하며, 가장 강한 누출 원천이다.

**Python이 Vibe에 미치는 긍정적 전이**:
- 들여쓰기 기반 블록 구조 (Vibe도 동일)
- `if`, `else`, `for`, `in` 키워드 (Vibe도 동일)
- `and`, `or`, `not` boolean 연산자 (Vibe도 동일)
- 함수 정의 시 `fn` ≈ `def` (유사성)

**Python이 Vibe에 미치는 부정적 전이**:
- `def` → Vibe는 `fn` (사소한 차이, 실제 오류 0건)
- `:` (블록 시작 전) → Vibe는 콜론 없음 (**오류 0건** -- 프롬프트로 방어 성공)
- `self` → Vibe는 명시적 매개변수 (실제 오류 0건)
- 미초기화 변수 (Python 힌트: `x: int`) → Vibe 실패 유발

**핵심 발견**: Vibe가 Python과 **공유하는** 패턴(들여쓰기, `and`/`or`/`not`)은 LLM 정확도를 높이고, **다른** 패턴(`:` 없음, `while` 없음)은 오류를 유발한다.

이것을 "패턴 전이 스펙트럼"이라고 부르자:

```
[긍정적 전이] <--------> [부정적 전이]
  공유 패턴                 차이 패턴
  (들여쓰기)               (while 없음)
  (and/or/not)             (: 없음)
  (if/else/for)            (초기화 필수)
```

**Vibe 설계의 전략적 위치**: Vibe는 의도적으로 Python과의 긍정적 전이를 극대화하고 부정적 전이를 최소화하는 위치에 설계되었다. 벤치마크 결과는 이 전략이 대체로 성공적임을 보여준다 (GPT-4o 90%, Claude 100%).

### 4.3 JavaScript/TypeScript의 영향

| JS/TS 패턴 | Vibe에서의 영향 | 관찰된 오류 |
|------------|----------------|-----------|
| `let`/`const` | 긍정적 전이 (Vibe도 동일) | 0 |
| `+=`, `-=` | 부정적 전이 | 6건 |
| `===` | 해당 없음 (Vibe는 `==`) | 0 |
| `null`/`undefined` | 해당 없음 (Vibe는 `none`) | 0 |
| `let x: Type;` (미초기화) | 부정적 전이 | 2건 |
| `{}` 블록 | 해당 없음 | 0 |

**`let`/`const` 공유의 효과**: Vibe가 JavaScript/TypeScript와 `let`/`const` 키워드를 공유하는 것은 매우 효과적인 설계 결정이다. LLM은 변수 선언 시 자연스럽게 `let`/`const`를 사용하며, 이로 인한 오류가 0건이다.

### 4.4 누출 방어 가능성 분류

프롬프트로 방어할 수 있는 패턴과 방어할 수 없는 패턴을 구분한다:

#### 프롬프트로 방어 가능 (Easy to enforce)

| 규칙 | 방어율 | 근거 |
|------|--------|------|
| `and`/`or`/`not` 사용 (`&&`/`||`/`!` 금지) | 100% | 3개 LLM 모두 성공 |
| 콜론 없는 블록 시작 | 100% | Python에서 `:` 누출 0건 |
| `fn` 사용 (`def` 금지) | 100% | 누출 0건 |
| 세미콜론 없음 | 100% | 누출 0건 |
| `draw_rect()`/`draw_circle()` API | 100% | API 호출은 프롬프트를 정확히 따름 |

#### 프롬프트로 방어 어려움 (Hard to enforce)

| 규칙 | 방어율 | 근거 |
|------|--------|------|
| `+=` 금지 (프롬프트에서 허용 시) | ~75% | 6/24 오류가 이 패턴 (프롬프트 오류 포함) |
| `while` 금지 | ~90% | GPT-4o만 위반, Claude/Gemini 성공 |
| 초기화 필수 | ~90% | GPT-4o만 위반 |

#### 프롬프트로 방어 불가능 (Impossible to enforce)

| 규칙 | 방어율 | 근거 |
|------|--------|------|
| 코드 절단 방지 | ~85% | Gemini 3건 절단, 모델 내부 문제 |
| 마크다운 혼입 방지 | ~85% | Gemini 2건 혼입, instruction following 문제 |

- Easy → Medium: 0% → 33% (33%p 증가)
- Medium → Hard: 33% → 80% (47%p 증가)

```

Medium (6 tasks): ████████████████████ 100%
Hard (10 tasks):  ████████████████████ 100%
```

Claude는 난이도와 무관하게 100%를 달성했다. Hard 태스크(breakout_game: ~67줄, chain_reaction: ~60줄)에서도 문법 오류가 0건이다.

### 5.5 핵심 발견: 복잡도-오류 관계

**발견 1**: 오류율은 복잡도에 대해 **비선형적(지수적)**으로 증가한다.
- 단순 태스크(< 50 토큰): 모든 LLM이 정확
- 중간 태스크(50-200 토큰): 약한 모델에서 실패 시작
- 복잡 태스크(200+ 토큰): 약한 모델은 대부분 실패, 강한 모델도 가끔 실패

**발견 2**: 복잡도 증가 시 오류 유형이 변화한다.
- 단순 태스크: 오류 거의 없음
- 중간 태스크: 절단 오류 (모델이 중간에 포기)
- 복잡 태스크: 패턴 누출 오류 (학습 데이터 패턴이 프롬프트를 override)

이것은 **"프롬프트 압력 vs 학습 데이터 압력"의 균형이 코드 길이에 따라 기울어진다**는 것을 의미한다. 짧은 코드에서는 프롬프트가 우세하지만, 긴 코드에서는 학습 데이터가 점차 우세해진다.

```
프롬프트 영향력  ■■■■■■■■■■■■■■■■■■■■  ← 짧은 코드
                 ■■■■■■■■■■■■■■■■        ← 중간 코드
                 ■■■■■■■■■■■■            ← 긴 코드
학습 데이터 영향력 ■■■■■                    ← 짧은 코드
                   ■■■■■■■■■■              ← 중간 코드
                   ■■■■■■■■■■■■■■■■        ← 긴 코드
```

**프로필**:
- Vibe 정확도: 90% (18/20)
- 실패 유형: 패턴 누출 (A1, A3, B1)

### 6.2 Claude Opus 4.6: "프롬프트를 가장 충실하게 따른다"

**프로필**:
- Vibe 정확도: 100% (20/20)
- 실패 유형: 없음
- 평균 토큰: 추정 ~170 (GPT-4o보다 약간 많음)

**Claude의 성공 메커니즘**:
Claude가 100%를 달성한 핵심 요인은 **"프롬프트 예제 패턴의 정밀한 모방"**이다. 프롬프트에 3개의 완전한 예제가 있었고, Claude는 이 예제의 패턴을 충실하게 따랐다:
- 변수 선언: `let x: Float = 400.0` (항상 초기화)
- 업데이트: `x = x + speed * dt` (명시적 재할당)
- 루프: `for i in range(0, n)` (range 기반)

## 7. 오류를 **방지하는** 문법 특성

벤치마크에서 **0건의 오류**를 유발한 Vibe 문법 특성을 분석하면, "LLM에 우호적인 설계"의 공통 원칙이 드러난다.

### 7.1 오류 0건인 문법 특성들

| 문법 특성 | 유사 언어 | 오류 건수 | 왜 안전한가? |
|-----------|----------|-----------|------------|
| `let`/`const` 변수 선언 | JavaScript/TypeScript | 0 | 가장 큰 학습 데이터와 일치 |
| `fn` 함수 선언 | Rust/Kotlin | 0 | 2글자, 모호하지 않음 |
| 들여쓰기 블록 | Python | 0 | Python 전이 극대화 |
| `if`/`else if`/`else` | 범용 | 0 | 모든 언어에서 동일 |
| `for i in range(a, b)` | Python | 0 | Python 패턴 그대로 |
| `and`/`or`/`not` | Python/SQL | 0 | Python과 동일 |
| `draw_rect(x, y, w, h)` | 게임 API 범용 | 0 | 직관적 이름 + 위치 인자 |
| `str()` 타입 변환 | Python | 0 | Python과 동일 |
| `List[Float] = []` 초기화 | Python 리스트 + 타입 | 0 | 패턴이 명확 |
| `xs = xs + [value]` 리스트 추가 | Python 연결 | 0 | 직관적 |
| `xs[i]` 인덱스 접근 | 범용 | 0 | 모든 언어에서 동일 |
| `-- comment` 주석 | Lua/SQL/Haskell | 0 | 단순하고 명확 |

### 7.2 공통 원칙 추출

**원칙 1: "최대 공약수" 설계**
오류가 없는 문법은 **여러 주요 언어의 교집합**에 해당한다. `if`, `else`, `for`, 인덱스 접근 `[]`, 함수 호출 `f(args)` 등은 거의 모든 언어에서 동일한 형태이므로 LLM이 틀릴 수 없다.

**원칙 2: "가장 큰 언어에 정렬"**
Python이 학습 데이터에서 가장 큰 비중을 차지하므로, Python과 공유하는 패턴(`and`/`or`/`not`, 들여쓰기, `for x in range()`, `str()`)은 LLM이 자연스럽게 올바르게 생성한다.

**원칙 3: "이름이 의미를 전달"**
`draw_rect`, `draw_circle`, `key_down`, `rand_float` 같은 함수 이름은 기능을 즉시 알 수 있다. LLM이 함수 이름만으로 시그니처를 정확하게 추론한다.

**원칙 4: "하나의 표현 방법"**
오류가 없는 패턴들은 모두 "이 개념을 표현하는 방법이 정확히 하나"인 경우다. 변수 선언은 `let x: Type = value`만 가능하고, 다른 방법이 없다.

### 7.3 오류를 **유발하는** 문법 특성

| 문법 특성 | 오류 건수 | 왜 위험한가? |
|-----------|-----------|------------|
| `+=` 미지원 | 6 | 거의 모든 언어에 존재하는 보편 패턴을 제거 |
| `while` 미지원 | 2 | 거의 모든 언어에 존재하는 키워드를 제거 |
| 초기화 필수 | 2 | 거의 모든 언어에서 미초기화 허용 |

**공통 원인**: "보편적으로 존재하는 것을 제거"하면 LLM이 프롬프트를 무시하고 학습 데이터 패턴을 사용한다.

## 8. Vibe 언어 설계에 대한 실행 가능한 권고사항

### 8.1 즉시 구현 권고 (프롬프트 또는 파서 수정)

#### R1: 복합 할당 연산자 구현 (우선순위: 최상)

**현상**: 전체 실패의 25%가 `+=` 관련
**조치**: `+=`, `-=`, `*=`, `/=`를 파서에 구현
**효과**: GPT-4o 정확도 90% → 100% (추정), Gemini 정확도 50% → 70% (추정)
**근거**: `x = x + n`은 LLM이 코드가 길어질수록 자동으로 `x += n`으로 전환하는 경향이 있다. 이를 방지하는 것보다 허용하는 것이 효율적이다.

#### R2: 미초기화 변수 선언 허용 (우선순위: 높음)

**현상**: GPT-4o 2건 실패
**조치**: `let x: Float`를 허용하되, 내부적으로 zero value(`0.0`, `0`, `""`, `false`, `[]`)로 초기화
**효과**: 조건부 초기화 패턴의 자연스러운 표현 허용
**근거**: Go의 zero value 패턴과 동일. TypeScript/Rust/Go 모두 미초기화 선언을 허용한다.

#### R3: 프롬프트에서 `+=` 관련 오류 수정 (우선순위: 즉시)

**현상**: vibe-context.ts에 `Assignment: = += -= *= /=`가 파서 미구현 상태에서 명시됨
**조치**: R1 구현 전까지 프롬프트에서 `+=` 관련 내용 제거, "use `x = x + n` instead of `x += n`" 명시
**효과**: 프롬프트 오류로 인한 실패 즉시 제거

### 8.2 중기 설계 권고 (언어 설계 수준)

#### R4: `while` 호환 레이어 (우선순위: 중간)

**선택지**:
- (a) `while`을 `for`의 별칭으로 허용 (파서에서 while → for 변환)
- (b) `while` 사용 시 명확한 에러 메시지: "use `for condition` instead of `while condition`"
- (c) `while` 완전 허용

**권장**: (a). `while`은 모든 프로그래머의 근육 기억에 새겨진 키워드다. 이를 거부하면 LLM과 인간 모두 반복적으로 실패한다. 별칭으로 허용하되, vibefmt에서 `for`로 정규화한다.

#### R5: 에러 메시지 LLM-friendly화 (우선순위: 중간)

현재 파서 에러: `expected EQ, got NEWLINE ("")`
개선: `Error: Variable declaration requires initialization. Use 'let x: Float = 0.0' instead of 'let x: Float'`

LLM이 에러 메시지를 피드백으로 받아 코드를 수정하는 시나리오에서, 구체적인 수정 방향을 제시하는 에러 메시지가 수정 성공률을 높인다.

#### R6: 프롬프트 예제 전략 강화 (우선순위: 높음)

**발견**: Claude 100% 성공의 핵심 요인은 "프롬프트 예제의 정밀한 모방"
**조치**: 시스템 프롬프트에 **복잡한 태스크의 예제도 포함** (현재는 단순한 예제 3개만)
**구체적 추가 예제**:
1. 리스트 순회 + 조건부 처리 (for + if 중첩)
2. 조건부 초기화 패턴 (`let x: Float = 0.0` + 분기별 덮어쓰기)
3. 상태 머신 패턴 (if/else if 체인)

### 8.3 장기 설계 원칙

#### P1: "보편적 패턴은 절대 제거하지 말 것"

**원칙**: `+=`, `while`, 미초기화 선언 등 **거의 모든 프로그래밍 언어에 존재하는 패턴**은 LLM의 학습 데이터 압력이 너무 강해서 프롬프트로 방어할 수 없다. 이런 패턴은 **허용하거나 호환 레이어를 제공**해야 한다.

**수치 근거**: 보편적 패턴 제거로 인한 실패 = 10/24 (42%). 전체 실패의 거의 절반이 "있어야 할 것이 없어서" 발생.

#### P2: "가장 큰 학습 데이터 언어에 정렬하라"

**원칙**: Python/JavaScript가 LLM 학습 데이터의 지배적 비중을 차지한다. Vibe의 문법은 가능한 한 이 두 언어와 겹치도록 설계하되, 독창적 문법이 필요한 경우 프롬프트 예제로 강하게 보강해야 한다.

#### P3: "문법이 아니라 의미가 같으면 된다"

**원칙**: `while condition` 과 `for condition`은 의미적으로 동일하다. LLM이 어느 쪽을 사용하든 같은 코드를 생성한다면, 파서가 양쪽 모두 수용하는 것이 100% 정확도에 유리하다.

## 9. "LLM 구문 오류 방지 점수표" (언어 설계 체크리스트)

새로운 문법 기능을 추가할 때, 아래 체크리스트로 LLM 우호성을 평가한다:

| 질문 | 점수 | 설명 |
|------|------|------|
| 이 문법은 Python에 존재하는가? | +3 | Python 전이를 받는다 |
| 이 문법은 JavaScript에 존재하는가? | +2 | JS 전이를 받는다 |
| 이 문법은 3개 이상의 주류 언어에 존재하는가? | +2 | 보편성이 높다 |
| 이 문법은 기존 키워드와 동일한 키워드를 사용하는가? | +1 | 키워드 학습 필요 없음 |
| 이 문법은 어떤 주류 언어에도 없는 완전히 새로운 것인가? | -3 | 프롬프트로만 학습해야 함 |
| 이 문법은 기존 보편 패턴을 금지하는가? | -5 | 학습 데이터 압력과 충돌 |
| 이 문법은 표현 방법이 유일한가? (한 가지 방법) | +2 | 비결정성 감소 |
| 이 문법의 에러 메시지가 수정 방향을 제시하는가? | +1 | LLM 자동 수정 지원 |

**점수 해석**:
- 5점 이상: LLM이 프롬프트만으로 정확하게 생성할 가능성 높음
- 0~4점: 프롬프트 예제가 필요. 벤치마크로 검증 권장
- 음수: LLM 생성 오류 위험. 재설계 또는 호환 레이어 필요

**Vibe의 기존 문법 평가 사례**:

| 문법 | 점수 | 상세 |
|------|------|------|
| `let x: Float = 0.0` | +8 | Python(+3), JS(+2), 3언어+(+2), 유일 방법(+2), 키워드 동일(-1) |
| `and`/`or`/`not` | +8 | Python(+3), SQL(+2), 3언어+(+2), 유일 방법(+2), 키워드(-1) |
| `for i in range(0, n)` | +7 | Python(+3), 2언어+(+2), 유일 방법(+2) |
| `while` 금지 → `for` 통합 | **-3** | 보편 패턴 금지(-5), 유일 방법(+2) |
| `+=` 금지 | **-3** | 보편 패턴 금지(-5), 유일 방법(+2) |
| 초기화 필수 | **-1** | 보편 패턴 금지(-5), 유일 방법(+2), Python 유사(+2) |

이 점수표가 벤치마크 결과를 정확히 예측한다: 점수가 음수인 3개 문법에서만 실패가 발생했다.

## 10. 결론

### 10.1 핵심 발견 요약

1. **LLM 코드 생성 오류의 42%는 "보편적 패턴 제거"에서 발생한다.** `+=`, `while`, 미초기화 선언 등 거의 모든 언어에 존재하는 패턴을 제거하면, LLM의 학습 데이터 압력이 프롬프트를 override한다.

2. **오류율은 코드 복잡도에 대해 비선형적(지수적)으로 증가한다.** 300+ 토큰의 복잡한 태스크에서 학습 데이터 패턴이 프롬프트를 압도하기 시작한다.

3. **가장 효과적인 LLM-friendly 설계 전략은 "최대 공약수"이다.** 여러 주요 언어의 교집합에 해당하는 문법을 채택하면, LLM이 틀릴 가능성이 원천적으로 제거된다.

4. **프롬프트 예제의 품질이 LLM 정확도를 결정한다.** Claude의 100% 성공은 프롬프트 예제 패턴의 정밀한 모방에서 기인했다.

5. **모델 간 instruction following 능력 차이가 언어 설계보다 더 큰 영향을 미친다.** Gemini는 Python(학습 데이터 최대)에서도 40%만 성공 -- 이는 언어 문제가 아니라 모델 문제다.

6. **학습 데이터 0인 Vibe가 GPT-4o 기준 90%에 도달한 것은 언어 설계의 유효성을 증명한다.** Python/Lua 대비 토큰 효율은 15~20% 우수하다.

### 10.2 Vibe 즉시 조치 사항

| 우선순위 | 조치 | 예상 효과 |
|---------|------|----------|
| **P0** | 프롬프트에서 `+=` 관련 오류 수정 | -6건 실패 |
| **P1** | `+=`, `-=`, `*=`, `/=` 파서 구현 | GPT-4o 100% 도달 |
| **P2** | 미초기화 변수 선언에 zero value 자동 부여 | GPT-4o 100% 확정 |
| **P3** | `while`을 `for`의 별칭으로 허용 | 패턴 누출 완전 제거 |

**예상**: P0~P3 적용 후 GPT-4o 100%, Gemini 65~70% 도달 가능. (Gemini의 30%는 절단/마크다운 등 모델 자체 문제로 언어 설계로는 해결 불가)

### 10.3 향후 연구 방향

1. **벤치마크 확장**: 20 → 50+ 태스크로 통계적 유의성 확보
2. **Claude 자동화**: 벤치마크에 Claude 정식 포함 (현재 수동)
3. **GDScript 추가**: 같은 게임 도메인 언어와의 직접 비교
4. **수정-재생성 루프**: LLM에게 에러 메시지를 피드백하고 재생성 시키는 실험
5. **프롬프트 최적화**: 예제 수/길이/복잡도 변화에 따른 정확도 변화 측정
6. **토큰 한계 실험**: 500, 1000, 2000 토큰 태스크에서의 오류율 변화 추적
