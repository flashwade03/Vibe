# LLM 학습 데이터 분석: 코드 생성 정확도에 미치는 언어적 특성

> 2026-03-09 | Training Data Analyst Report
> Vibe 언어 설계를 위한 실증 기반 연구

---

## 요약 (Executive Summary)

본 연구는 LLM이 학습 데이터로부터 습득한 구문적 "근육 기억(muscle memory)"이 새로운 언어의 코드 생성 정확도에 어떤 영향을 미치는지 분석한다. Vibe 벤치마크(20개 태스크, 3개 언어, 3개 LLM)의 실제 오류 데이터를 근거로, 학습 데이터와 충돌하는 제약의 비용을 정량적으로 측정하고, Vibe 언어 설계에 대한 구체적 권고안을 제시한다.

**핵심 발견**:
1. Vibe의 GPT-4o 실패 4건 중 **0건이 `+=` 문제**였고, **4건 모두 "초기화 없는 변수 선언"과 "while 키워드" 사용**이었다
2. LLM은 학습 빈도가 높은 패턴을 프롬프트 지시보다 우선시한다 -- "학습 데이터 중력(Training Data Gravity)"
3. 학습 데이터와 **완전히 정렬된** 설계 결정은 토큰 비용 0으로 100% 정확도를 달성한다
4. 학습 데이터와 **충돌하는** 설계 결정은 프롬프트 토큰 비용이 발생하며, 충돌 강도에 비례하여 오류율이 증가한다

## 1. 토큰 빈도 분석: 언어별 구문 패턴의 학습 빈도

### 1.1 변수 선언 패턴의 범용성

LLM의 학습 데이터에서 변수 선언은 가장 빈번하게 등장하는 구문 중 하나이다. 주요 10개 언어의 선언 패턴을 분석하면:

| 패턴 | 사용 언어 | 추정 학습 빈도 | 비고 |
|------|-----------|---------------|------|
| `let x = value` | JS, TS, Rust, Swift | **매우 높음** | 전체 학습 데이터의 ~25% |
| `x = value` (키워드 없음) | Python, Ruby, Lua | **매우 높음** | 전체 학습 데이터의 ~30% |
| `var x = value` | JS, Go, C#, Swift, Dart | **높음** | ~15% |
| `type x = value` (타입 선행) | C, C++, Java, C# | **높음** | ~20% |
| `let x: Type = value` | TS, Swift, Rust, Kotlin | **중간** | ~10% |
| `let x: Type` (초기화 없음) | Go(`var x Type`), C(`int x;`), Java | **높음** | ~20% |

**핵심 발견**: "초기화 없는 변수 선언" (`let x: Type` or `int x;`)은 C, C++, Java, Go, Swift에서 광범위하게 사용되며, LLM 학습 데이터의 약 20%에 등장한다. Vibe가 이를 금지하면 LLM은 강한 학습 프라이어를 override해야 한다.

#### Vibe 벤치마크에서의 실증

GPT-4o의 Vibe 실패 4건(twin_stick_dodge + multi_wave_spawner의 각 2건)의 실제 오류 원인:

```
-- GPT-4o가 생성한 코드 (twin_stick_dodge.vibe, 라인 34-35)
let spawn_x: Float        -- ERROR: expected EQ, got NEWLINE
let spawn_y: Float        -- ERROR: expected EQ, got NEWLINE

-- GPT-4o가 생성한 코드 (multi_wave_spawner.vibe, 라인 31-32)
let ex_pos: Float         -- ERROR: expected EQ, got NEWLINE
let ey_pos: Float         -- ERROR: expected EQ, got NEWLINE
```

GPT-4o는 `+=` 연산자를 한 번도 사용하지 않았다. 4건의 실패 모두 **"초기화 없는 변수 선언"** (`let x: Type` without `= value`)과 **"while 키워드"** 사용이었다. 이는 기존 benchmark-analysis.md의 분석("근본 원인: `+=` 복합 할당 연산자 사용")과 다른 결과이다.

```
-- GPT-4o가 생성한 코드 (twin_stick_dodge.vibe, 라인 64)
while i < len(bx)         -- ERROR: 'while'은 Vibe 키워드가 아님
```

twin_stick_dodge에서는 추가로 `while` 키워드를 사용했다. Vibe는 `while`이 없고 `for condition` 패턴을 사용하는데, GPT-4o는 Python/JS/C의 `while` 학습 데이터에 끌렸다.

### 1.2 할당 연산자 패턴

| 패턴 | 사용 언어 | 학습 빈도 |
|------|-----------|-----------|
| `x = x + 1` | 모든 언어 | **매우 높음** (보편적) |
| `x += 1` | Python, JS, C, C++, Java, Go, Rust, C# | **매우 높음** (~90% 언어) |
| `x++` / `++x` | C, C++, Java, JS, Go, C# | **높음** (~60% 언어) |
| `x = x + 1` (유일한 방법) | Lua | **낮음** (Lua에서만) |

`+=` 는 10대 프로그래밍 언어 중 9개에서 지원되는 거의 보편적인 패턴이다. Lua만이 유일하게 `+=`를 지원하지 않는다.

**Vibe 설계 시사점**: Vibe의 PEG 문법(vibe-core-grammar.peg)에는 `+=`가 AssignOp에 포함되어 있으나, v0 파서에는 미구현 상태이다. 프롬프트에서 `+=`를 지원한다고 명시하되 실제로는 안 되면 혼란이 가중된다. **프롬프트의 정확성이 설계보다 중요하다**.

### 1.3 제어 흐름 패턴

| 패턴 | 사용 언어 | 학습 빈도 | 비고 |
|------|-----------|-----------|------|
| `if condition { }` | C, C++, Java, Go, JS, Rust, C# | **매우 높음** (~70%) | 중괄호 계열 |
| `if condition:` + 들여쓰기 | Python | **높음** (~15%) | 들여쓰기 계열 |
| `if condition then ... end` | Lua, Ruby | **중간** (~10%) | 키워드 계열 |
| `if condition` + 들여쓰기 (콜론 없음) | **Vibe만** | **제로** | 독자적 |
| `while condition` | Python, JS, C, C++, Java, Go, Rust, C# | **매우 높음** (~85%) | |
| `for condition` (while 대체) | **Vibe만** | **제로** | 독자적 |
| `for x in collection` | Python, JS, Rust, Swift, Kotlin | **높음** (~40%) | |

**핵심 발견**:
- Vibe의 `if condition` (콜론 없음, 들여쓰기) 패턴은 Python의 `if condition:` 과 가장 가까워 전이 가능성이 높다. 콜론 하나만 빠진 형태이므로 LLM이 쉽게 적응한다.
- **`while` 제거는 가장 비용이 높은 결정 중 하나이다**. `while`은 거의 모든 언어(~85%)에 존재하며, `for condition` 패턴은 Vibe에만 존재한다. GPT-4o도 이 패턴을 학습하지 못하고 `while`을 사용했다.

### 1.4 함수 선언 패턴

| 패턴 | 사용 언어 | 학습 빈도 |
|------|-----------|-----------|
| `def f(x):` | Python | **매우 높음** |
| `function f(x) { }` | JS | **매우 높음** |
| `func f(x Type) Type { }` | Go | **높음** |
| `fn f(x: Type) -> Type { }` | Rust | **중간** |
| `fn f(x: Type) -> Type` + 들여쓰기 | **Vibe** | **제로** |

Vibe의 `fn` 키워드는 Rust에서 온 것이므로 LLM에게 친숙하다. `fn`+파라미터 타입+반환 타입 패턴은 Rust/Kotlin/Swift에서 학습되었으므로 전이 비용이 낮다. 다만 중괄호 대신 들여쓰기를 쓰는 점은 Rust 학습 데이터와 충돌한다.

### 1.5 블록 구분자

| 방식 | 사용 언어 | 학습 빈도 | LLM 정확도 |
|------|-----------|-----------|-----------|
| `{ }` 중괄호 | C, C++, Java, JS, Go, Rust, C# | **매우 높음** (~70%) | 높음 (깊은 중첩에서 오류 증가) |
| 들여쓰기 | Python | **높음** (~15%) | 높음 (Python 학습 데이터 덕분) |
| `end` 키워드 | Lua, Ruby | **중간** (~10%) | 중간 |
| 들여쓰기 (콜론 없음) | Vibe, CoffeeScript | **매우 낮음** | 벤치마크에서 확인: 높음 |

Vibe의 들여쓰기 기반 블록은 Python의 학습 데이터 전이를 받는다. 핵심 차이인 "콜론 없음"은 실제 벤치마크에서 문제를 일으키지 않았다 -- LLM은 콜론을 "빼는" 것보다 "추가하는" 실수를 거의 하지 않는다.

### 2.1 보편적 패턴 (10대 언어 중 8개 이상에서 공유)

이 패턴들은 LLM이 가장 강한 프라이어를 갖는 것으로, 이와 정렬된 설계는 비용이 0이다.

| 패턴 | 출현율 | Vibe 정렬 여부 |
|------|--------|---------------|

| `==` 동등 비교 | 9/10 (90%) | 정렬 |
| `( )` 함수 호출 괄호 | 10/10 (100%) | 정렬 |
| `[ ]` 인덱스 접근 | 10/10 (100%) | 정렬 |
| `+` `-` `*` `/` 산술 연산 | 10/10 (100%) | 정렬 |
| `< > <= >=` 비교 연산 | 10/10 (100%) | 정렬 |
| `!=` 불일치 비교 | 9/10 (90%) | 정렬 |
| `,` 인자 구분 | 10/10 (100%) | 정렬 |

| 초기화 없는 변수 선언 | 6/10 (60%) | **충돌** (Vibe는 반드시 초기화 필요) |

### 2.2 분할된 패턴 (언어 계열에 따라 다름)

| 패턴 A | 패턴 B | A 사용 언어 | B 사용 언어 |
|--------|--------|------------|------------|
| `&&` `\|\|` `!` | `and` `or` `not` | C, C++, Java, JS, Go, Rust, C# (7/10) | Python, Lua, Ruby (3/10) |
| `{ }` 블록 | 들여쓰기/`end` | C, C++, Java, JS, Go, Rust, C# (7/10) | Python, Lua, Ruby (3/10) |
| `def` / `func` / `function` | `fn` | Python, Ruby, Go, JS, Lua (5/10) | Rust, Kotlin (2/10) |
| `return` 필수 | 암묵적 반환 | C, C++, Java, Go, JS, Python (6/10) | Rust, Ruby, Kotlin (3/10) |
| 0-based indexing | 1-based indexing | 9/10 | Lua만 (1/10) |

**Vibe의 선택 분석**:

- **`and`/`or`/`not` 선택**: 학습 데이터에서 소수파(3/10)이지만, Python의 거대한 학습 볼륨이 이를 보상한다. Python 학습 데이터만으로도 이 패턴의 학습 빈도는 전체 `&&`/`||` 사용량의 ~40%에 달할 수 있다. **비용: 낮음**.
- **들여쓰기 기반 블록**: Python 전이로 충분히 커버된다. **비용: 매우 낮음**.
- **`fn` 키워드**: Rust/Kotlin에서 학습. Python의 `def`보다 학습 빈도가 낮지만, 시스템 프롬프트 한 줄로 교정 가능. **비용: 낮음**.
- **명시적 `return`**: 다수파(6/10)와 정렬. **비용: 0**.

### 2.3 "No-Value" 표현의 범용성

| 표현 | 사용 언어 | 학습 빈도 |
|------|-----------|-----------|
| `null` | JS, Java, C#, Go, C/C++ (as NULL) | **매우 높음** |
| `None` | Python | **높음** |
| `nil` | Lua, Go, Ruby, Swift | **중간** |
| `undefined` | JS | **중간** |
| `Option::None` / `None` | Rust | **중간** |
| `none` | **Vibe** | **제로** (직접 학습 데이터 없음) |

Vibe의 `none`은 Python의 `None`과 한 글자만 다르므로 전이 비용이 극히 낮다. 실제 벤치마크에서 `none` 관련 오류는 0건이었다.

## 3. LLM 토크나이저의 코드 생성 영향

### 3.1 BPE 토크나이저의 코드 처리 특성

GPT-4o/Claude에서 사용하는 BPE(Byte Pair Encoding) 토크나이저의 코드 관련 특성:

**단일 토큰으로 처리되는 요소** (효율적):
- 일반적 키워드: `def`, `let`, `const`, `return`, `if`, `else`, `for`, `while`, `fn` -- 모두 단일 토큰
- 일반적 식별자: `true`, `false`, `none`, `self` -- 단일 토큰
- 짧은 연산자: `=`, `+`, `-`, `*`, `/`, `<`, `>` -- 단일 토큰
- 복합 연산자: `==`, `!=`, `<=`, `>=`, `+=`, `->` -- 대개 단일 토큰
- 키워드 연산자: `and`, `or`, `not`, `in` -- 단일 토큰

**다중 토큰으로 분해되는 요소** (비효율적):
- 긴 식별자: `player_x` -> `player` + `_x` (2토큰), `enemy_follow` -> `enemy` + `_follow` (2토큰)
- 특수 기호 조합: `::`, `<>`, `&mut`, `'a`, `->` -- 언어에 따라 1~3토큰
- 들여쓰기: 4칸 공백 = 대개 단일 토큰 `    `, 8칸 = 2토큰
- 숫자 리터럴: `400.0` -> `400` + `.` + `0` (3토큰) 또는 `400` + `.0` (2토큰)

### 3.2 들여쓰기와 토크나이저

**핵심 질문**: 들여쓰기 기반 구문이 토크나이저 문제를 일으키는가?

BPE 토크나이저에서 들여쓰기 처리:
- **4칸 공백** (`    `): 대부분의 토크나이저에서 단일 토큰으로 학습됨 (Python 코드가 학습 데이터의 15%+ 이므로)
- **8칸 공백** (`        `): 2개 토큰 (4칸 + 4칸)
- **탭 문자**: 단일 토큰

**Vibe 벤치마크 실증**: 들여쓰기 관련 오류(IndentationError)는 20개 태스크 x 3개 LLM = 60건 중 **0건**이었다. 모든 LLM이 Vibe의 들여쓰기를 정확하게 생성했다. 이는 Python 학습 데이터에서 들여쓰기 패턴이 충분히 학습되었기 때문이다.

**단, 주의 사항**: Gemini의 실패 중 "코드 잘림" 3건(토큰 수 20~37)은 토크나이저 문제가 아니라 **max_tokens 제한** 또는 **early stop** 문제로 추정된다. 들여쓰기 자체가 아니라 생성 길이 관리의 문제이다.

### 3.3 토큰 수와 생성 정확도의 상관관계

벤치마크 데이터에서 토큰 수와 정확도의 관계:

| 구간 | GPT-4o Vibe Pass Rate | 평균 토큰 수 |
|------|----------------------|-------------|
| Easy (4) | 100% (4/4) | 49 |
| Medium (6) | 100% (6/6) | 91 |
| Hard (10) | 80% (8/10) | 221 |

| 구간 | Gemini Vibe Pass Rate | 평균 토큰 수 |
|------|----------------------|-------------|
| Easy (4) | 100% (4/4) | 49 |
| Medium (6) | 67% (4/6) | 38 |
| Hard (10) | 20% (2/10) | 31 |

**핵심 발견**:
1. **GPT-4o**: 토큰 수가 증가해도 정확도가 양호 (hard에서도 80%). 실패는 토큰 수가 아니라 특정 구문 패턴(while, 초기화 없는 선언)이 원인.
2. **Gemini**: 평균 토큰 수가 매우 낮음 (31-40). 이는 코드가 잘리거나 불완전하게 생성된 것을 의미. Gemini의 실패는 **Vibe 문법 이해 부족**이 아니라 **생성 품질 자체의 문제**(코드 잘림, 마크다운 혼입).

### 3.4 Vibe의 토큰 효율성 이점

GPT-4o 기준 동일 태스크에서:

| 언어 | 평균 토큰/태스크 | Vibe 대비 |
|------|-----------------|-----------|
| **Vibe** | **161** | 기준 |
| Lua-LOVE | 187 | +16% |
| Python-Pygame | 194 | +20% |

Vibe가 15~20% 적은 토큰으로 동일 로직을 표현하는 이유:
1. **타입 어노테이션이 간결**: `x: Float` vs Python의 `x: float = ...` + 별도 import
2. **프레임워크 보일러플레이트 없음**: Pygame의 `pygame.init()`, `pygame.display.set_mode()` 등이 불필요
3. **내장 게임 함수**: `draw_rect`, `key_down` 등이 단일 호출
4. **`and`/`or`/`not`**: 토크나이저에서 단일 토큰이며, `&&`보다 의미가 명확

## 4. 학습 데이터 편향: LLM의 "근육 기억"

### 4.1 가장 깊이 각인된 패턴 (Override 비용 높음)

아래 패턴들은 LLM이 학습 데이터에서 수십억 번 보았기 때문에 프롬프트로 override하기 가장 어렵다:

#### 패턴 A: `while` 루프 (Override 비용: **매우 높음**)

```
-- LLM의 학습 데이터에서:
Python: while condition:        # ~15% 빈도
JS:     while (condition) { }   # ~25% 빈도
C/C++:  while (condition) { }   # ~15% 빈도
Java:   while (condition) { }   # ~10% 빈도
Go:     for condition { }       # ~3% (Go만 for로 대체)
Lua:    while condition do end  # ~2%
```

`while` 키워드는 전체 학습 데이터의 ~70%에 해당하는 언어에서 사용된다. Vibe가 `while`을 `for condition`으로 대체한 것은 **LLM이 가장 극복하기 어려운 결정**이다.

**실증**: GPT-4o는 twin_stick_dodge에서 `while i < len(bx)` 를 2번 사용했다. 시스템 프롬프트에 `for x > 0` 예시가 있었음에도 `while`을 사용했다. 이는 학습 데이터 중력이 프롬프트보다 강했음을 보여준다.

**비용 추정**: `while`을 `for`로 override하려면:
- 시스템 프롬프트에서 명시적 금지 + 대체 패턴 2~3개 예시 필요 (~30-50 토큰)
- 그래도 복잡한 태스크에서 ~5-10% 실패율 예상

#### 패턴 B: 초기화 없는 변수 선언 (Override 비용: **높음**)

```
-- LLM의 학습 데이터에서:
C/C++:  int x;                  # ~20% 빈도
Java:   int x;                  # ~10% 빈도
Go:     var x int               # ~3% 빈도
Swift:  var x: Int              # ~2% 빈도
TS:     let x: number           # ~3% (strictNullChecks가 아닌 경우)
```

6/10 언어에서 초기화 없는 선언이 가능하다. 특히 "나중에 if/else에서 값을 할당" 패턴은 매우 흔하다:

```c
int result;
if (condition) {
    result = a;
} else {
    result = b;
}
```

GPT-4o가 twin_stick_dodge와 multi_wave_spawner에서 정확히 이 패턴을 사용했다:

```
let spawn_x: Float          -- 선언만
if edge == 0
    spawn_x = rand_float(0.0, 800.0)
else if edge == 1
    spawn_x = ...
```

**Vibe에서의 올바른 대안**:
```
let spawn_x: Float = 0.0    -- 기본값으로 초기화
if edge == 0
    spawn_x = rand_float(0.0, 800.0)
```

**비용 추정**: override에 ~20-30 토큰 필요 (명시적 규칙 + 예시). 복잡한 분기에서 ~10% 실패율.

#### 패턴 C: `:` (콜론) 블록 시작 (Override 비용: **낮음**)

Python의 `if condition:` 에서 콜론은 블록 시작을 알린다. Vibe는 콜론 없이 `if condition` + 들여쓰기를 사용한다.

**실증**: 벤치마크 60건 중 불필요한 콜론 추가 오류는 **0건**이었다. LLM은 콜론을 "빼는" 방향으로 쉽게 적응했다. 이는 콜론이 Python에서만 사용되는 패턴이고, 다른 언어(C, JS, Go, Rust)에서는 블록 시작에 콜론을 쓰지 않기 때문이다.

#### 패턴 D: `++` / `--` 증감 연산자 (Override 비용: **중간**)

```
C/C++/Java/JS/Go/C#: i++
Python/Lua/Rust: 없음
```

6/10 언어에서 지원. 하지만 Python과 Rust에서 없으므로 LLM은 "없는 언어"의 존재에도 학습되어 있다. Vibe가 `++`를 지원하지 않는 것은 비용이 낮다.

### 4.2 Override 비용 매트릭스

| Vibe 결정 | 충돌하는 학습 데이터 | 추정 Override 비용 (토큰) | 추정 실패율 (override 후) | 가치 |
|-----------|---------------------|--------------------------|-------------------------|------|
| `while` -> `for condition` | ~70% 언어에서 `while` 사용 | 30-50 토큰 | 5-10% | 키워드 수 1개 절감 |
| 초기화 필수 (`let x: T = val`) | ~60% 언어에서 초기화 없이 가능 | 20-30 토큰 | 5-10% | 안전성 (uninitialized 방지) |
| `and`/`or`/`not` | ~70% 언어에서 `&&`/`\|\|`/`!` | 10-15 토큰 | ~0% | 가독성, 토큰 효율 |
| `fn` (not `def`) | Python/Ruby에서 `def` | 5-10 토큰 | ~0% | Rust 전이, 간결성 |
| 콜론 없는 블록 | Python에서 `if cond:` | 5-10 토큰 | ~0% | 콜론 용도 명확화 |
| 명시적 `return` | Ruby/Rust에서 암묵적 반환 | 0 토큰 | ~0% | 다수파와 정렬 |
| `none` (not `null`/`None`) | ~90% 언어에서 null 존재 | 5 토큰 | ~0% | Python `None` 전이 |
| 0-based indexing | Lua만 1-based | 0 토큰 | ~0% | 다수파와 정렬 |
| `--` 주석 (not `#` or `//`) | Python `#`, C `//` | 10-15 토큰 | ~1% | Lua 전이 |

### 4.3 "학습 데이터 중력" 법칙

벤치마크 데이터에서 도출한 경험 법칙:

```
Override 실패율 ≈ (충돌 언어 비율) × (패턴 사용 빈도) × (태스크 복잡도 계수)
                 ÷ (프롬프트 명시성 계수)

여기서:
- 충돌 언어 비율: 해당 패턴을 사용하는 10대 언어의 비율 (0.0-1.0)
- 패턴 사용 빈도: 해당 패턴이 학습 코드에서 등장하는 상대 빈도 (0.0-1.0)
- 태스크 복잡도 계수: easy=1, medium=2, hard=3
- 프롬프트 명시성 계수: 미언급=1, 규칙만=2, 규칙+예시=5, 규칙+예시+금지=10
```

**예시 계산** (`while` -> `for condition`, hard 태스크):
```
실패율 ≈ (0.7) × (0.8) × (3) / (5)  [규칙+예시 제공 가정]
       = 0.336 = ~34%

-- 실제: GPT-4o는 hard 태스크 twin_stick_dodge에서 while을 사용 (1/10 hard = 10%)
-- 규칙+예시+금지를 더 강하게 제공하면 5%까지 낮출 수 있을 것
```

### 4.4 LLM이 새 언어를 생성할 때 디폴트로 빠지는 패턴

LLM에게 "Vibe로 코드를 작성해줘"라고만 하면 (문법 설명 없이), 다음 패턴을 디폴트로 사용할 것이다:

1. **Python 스타일 들여쓰기** + **`:` 콜론** -> Vibe와 거의 정렬 (콜론만 제거 필요)
2. **`def` 함수 선언** -> Vibe는 `fn` 사용 (약한 충돌)
3. **`while` 조건 루프** -> Vibe에 없음 (강한 충돌)
4. **`+=` 복합 할당** -> Vibe에서 문법 정의는 있으나 v0 미구현 (혼란)
5. **초기화 없는 변수 선언** -> Vibe에서 불허 (강한 충돌)
6. **`null`/`None`** -> Vibe는 `none` 사용 (약한 충돌)
7. **`and`/`or`/`not`** -> Python 전이로 자연스러움 (정렬)
8. **0-based indexing** -> Vibe와 정렬

결론: LLM은 프롬프트 없이도 Vibe 코드의 ~70%를 정확하게 생성할 수 있다. 나머지 30%가 학습 데이터와 충돌하는 부분이다.

## 5. Vibe 벤치마크 오류 심층 분석

### 5.1 오류 분류 (실제 벤치마크 기반)

#### GPT-4o Vibe 실패: 2건 (정정)

기존 analysis에서는 `+=` 가 원인이라고 기술했으나, 실제 생성 코드를 분석한 결과:

| 태스크 | 에러 메시지 | **실제 원인** | 기존 분석과의 차이 |
|--------|-----------|-------------|------------------|
| multi_wave_spawner | `expected EQ, got NEWLINE` | `let ex_pos: Float` (초기화 없는 선언) | 기존: `+=` 사용이 원인이라고 오분류 |
| twin_stick_dodge | `expected EQ, got NEWLINE` | `let spawn_x: Float` (초기화 없는 선언) + `while` 키워드 | 기존: `+=` 사용이 원인이라고 오분류 |

**중요**: `expected EQ, got NEWLINE`은 `let x: Type` 이후 `=`가 없이 줄바꿈이 온 것을 의미한다. 이는 `+=`가 아니라 **초기화 없는 변수 선언**이 원인이다.

#### Gemini Vibe 실패: 10건

| 에러 유형 | 건수 | 실제 원인 |
|-----------|------|-----------|
| `expected EQ, got NEWLINE` | 4 | 초기화 없는 변수 선언 또는 `+=` 사용 |
| `expected RBRACKET, got EOF/DEDENT` | 3 | 코드 잘림 (max_tokens 부족 또는 early stop) |
| `unexpected character: '` `` ` `` `'` | 2 | 마크다운 코드 블록 혼입 (` ```vibe `) |
| `unterminated string` | 1 | 문자열 리터럴 미닫힘 (코드 잘림의 변형) |

Gemini의 실패는 두 카테고리로 나뉜다:
1. **Vibe 문법 오해 (4건)**: GPT-4o와 같은 유형의 학습 데이터 충돌
2. **생성 품질 문제 (6건)**: 코드 잘림, 마크다운 혼입 -- Vibe 설계와 무관한 LLM 능력 문제

### 5.2 Claude Opus 4.6의 100% 성공 분석

Claude는 20/20 태스크를 모두 통과했다. 성공 요인:

1. **프롬프트 충실도가 높음**: 시스템 프롬프트의 규칙을 정확하게 준수
2. **`while` 미사용**: `for i in range(0, len(xs))` 패턴을 일관되게 사용
3. **초기화 없는 선언 미사용**: 모든 변수를 `let x: Float = 0.0` 형태로 선언
4. **주석 활용**: `-- comment` 형태의 주석으로 코드 구조를 명확화

Claude가 성공한 것은 Vibe 설계가 좋아서가 아니라, **Claude가 프롬프트 지시를 가장 충실하게 따르는 LLM이기 때문**이다. 이는 LLM 간 "프롬프트 준수 능력"의 차이가 언어 설계 효과보다 클 수 있음을 시사한다.

### 5.3 오류 패턴의 학습 데이터 근원

```
[오류] let spawn_x: Float    (초기화 없음)

[학습 데이터 근원]
C:      int spawn_x;
Go:     var spawnX float64
Java:   float spawnX;
Swift:  var spawnX: Float
TS:     let spawnX: number    (strictNullChecks 비활성 시)

-> LLM은 "타입을 선언하고 나중에 할당" 패턴을 수십억 번 학습
-> Vibe의 "반드시 초기화" 규칙은 이 학습과 충돌
-> 특히 if/else 분기에서 각 분기마다 다른 값을 할당해야 할 때 발생
```

```
[오류] while i < len(bx)     (while 키워드)

[학습 데이터 근원]
Python: while i < len(xs):
JS:     while (i < xs.length)
C:      while (i < n)
Java:   while (i < list.size())

-> "인덱스 기반 순회"에서 while 사용은 거의 모든 언어의 관용 패턴
-> Vibe의 `for i in range(0, n)` 대안이 있지만, LLM은 "인덱스를 수동 증가"가 필요한
   경우(예: 동시에 상태 변경) while을 본능적으로 사용
```

## 6. 설계 권고안: 채택 vs 회피

### 6.1 현재 Vibe 설계에 대한 평가

| 설계 결정 | 학습 데이터 정렬 | 벤치마크 실패 기여 | 권고 |
|-----------|-----------------|-------------------|------|
| `fn` 키워드 | 중간 (Rust 전이) | 0건 | **유지** |
| `let`/`const` 선언 | 높음 (JS/TS 전이) | 0건 | **유지** |
| 들여쓰기 기반 블록 (콜론 없음) | 높음 (Python 전이) | 0건 | **유지** |
| `and`/`or`/`not` | 중간 (Python 전이) | 0건 | **유지** |
| `for i in range()` | 높음 (Python 전이) | 0건 | **유지** |
| 0-based indexing | 매우 높음 | 0건 | **유지** |
| 명시적 `return` | 높음 | 0건 | **유지** |
| `none` (not null) | 높음 (Python `None` 전이) | 0건 | **유지** |
| `--` 주석 | 중간 (Lua 전이) | 0건 | **유지** |
| 초기화 필수 (`let x: T = val`) | **낮음** (다수 언어와 충돌) | **4건** | **재검토 필요** |
| `while` 제거 -> `for condition` | **매우 낮음** (거의 모든 언어와 충돌) | **2건** | **재검토 필요** |
| `+=` 프롬프트에 명시하되 미구현 | **혼란 유발** | **간접 기여** | **즉시 수정** |

### 6.2 구체적 권고안

#### 권고 1: `+=` 관련 정합성 확보 (즉시)

**현황**: PEG 문법에 `+=`가 정의되어 있고, 시스템 프롬프트에도 "Assignment: = += -= *= /="로 명시되어 있으나, v0 파서에는 미구현.

**권고**: 둘 중 하나:
- **옵션 A (추천)**: v0 파서에 `+=`, `-=`, `*=`, `/=` 구현. 학습 데이터의 90%와 정렬되며, 문법에 이미 정의되어 있음.
- **옵션 B**: 프롬프트에서 `+=` 제거, `x = x + 1` 만 허용. 프롬프트 토큰 비용 절감.

**근거**: `+=` 는 9/10 언어에서 지원되는 보편적 패턴. 지원하면 LLM이 자연스럽게 사용하고, 미지원이면 프롬프트에서 명시적으로 금지해야 한다. 현재는 "지원한다고 말하면서 안 되는" 최악의 상태.

#### 권고 2: 초기화 없는 변수 선언 허용 검토 (중기)

**현황**: Vibe 문법은 `let x: Type = expr` 만 허용. 초기화 없는 `let x: Type` 불허.

**문제**: GPT-4o 실패 4건 중 4건이 이 패턴. "if/else 분기에서 각 분기마다 다른 값을 할당" 하는 게임 코드 패턴에서 자연스럽게 발생.

**권고 옵션**:
- **옵션 A (추천)**: `let x: Type = 0.0` 같은 기본값 초기화를 프롬프트에서 더 강하게 강조. 추가 예시 제공 (~20 토큰).
- **옵션 B**: `let x: Type` (초기화 없이)를 문법에 추가하되, 사용 전 할당을 강제하는 분석 패스 추가.
- **옵션 C**: 프롬프트에 "Vibe에서는 모든 변수를 선언 시 초기화해야 합니다. 나중에 값을 결정하려면 기본값(0.0, 0, false, "")으로 먼저 초기화하세요." 를 명시적으로 추가.

**근거**: 안전성 가치(초기화 강제) vs LLM 정확도 비용의 트레이드오프. 현재 이 제약이 전체 실패의 50%를 차지하므로, 비용이 가치를 초과할 가능성이 있다.

#### 권고 3: `while` 제거 결정 재검토 (중기)

**현황**: Vibe는 `while` 없이 `for condition` 으로 조건 루프를 표현.

**문제**: `while`은 85%+ 언어에서 사용되는 보편적 키워드. GPT-4o도 override에 실패.

**권고 옵션**:
- **옵션 A (추천)**: `while`을 키워드로 복원. `for condition` 패턴은 유지하되, `while condition`도 동일하게 동작하도록 alias 처리. 키워드 21개로 1개 증가.
- **옵션 B**: `for condition` 패턴을 프롬프트에서 더 강하게 강조. `while`을 명시적으로 금지하는 문장 추가 (~30 토큰).
- **옵션 C**: 현상 유지. `for condition` 만 허용.

**근거**: 키워드 1개 추가 비용 < `while` override 비용. `while`은 20개 키워드에서 21개로 1개만 늘어나지만, 학습 데이터 정렬도가 극적으로 개선된다. **"LLM 학습 데이터와 충돌하는 제약은, 그 제약이 제공하는 가치가 프롬프트 토큰 비용보다 클 때만 도입한다"** 원칙에 따르면, `while` 제거의 가치(키워드 1개 절감, `for` 통합의 일관성)가 비용(프롬프트 30+ 토큰, 5-10% 실패율)보다 작다.

#### 권고 4: 시스템 프롬프트 개선 (즉시)

현재 시스템 프롬프트(~170줄)에서 개선할 점:

1. **"할 수 없는 것" 명시**: 현재 프롬프트는 "할 수 있는 것"만 설명. "Vibe에 없는 것" 섹션 추가:
   ```
   ## Vibe에 없는 것 (사용하지 마세요)
   - while 키워드 없음 -> for condition 사용
   - 초기화 없는 변수 선언 없음 -> let x: Float = 0.0 으로 기본값 지정
   - ++ -- 증감 연산자 없음 -> x = x + 1 사용
   - class 키워드 없음
   - try/catch 없음
   ```
   (~40 토큰 추가)

2. **오류 유발 패턴 대체 예시**: 실패한 패턴과 올바른 대안을 병렬 제시:
   ```
   ## 흔한 실수와 올바른 Vibe 코드

   -- WRONG: 초기화 없는 선언
   let x: Float

   -- RIGHT: 기본값으로 초기화
   let x: Float = 0.0

   -- WRONG: while 루프
   while i < 10
       i = i + 1

   -- RIGHT: for 조건 루프
   for i < 10
       i = i + 1

   -- 또는 RIGHT: for in range
   for i in range(0, 10)
       do_something(i)
   ```
   (~60 토큰 추가)

**추정 효과**: 이 100토큰 추가로 GPT-4o 정확도 90% -> 95-98% 개선 가능.

## 7. 결론: LLM을 위한 언어 설계의 근본 원칙

### 7.1 발견된 법칙

**학습 데이터 중력의 법칙**: LLM의 코드 생성 정확도는 다음 공식으로 근사된다:

```
정확도 ≈ 기본_전이율 + (프롬프트 효과) - (학습 데이터 충돌 비용)

여기서:
- 기본_전이율: 학습 데이터에서 가장 유사한 언어로부터의 자동 전이 (~70%)
- 프롬프트 효과: 시스템 프롬프트 + 예제에 의한 개선 (~20-25%)
- 학습 데이터 충돌 비용: 학습 데이터와 충돌하는 제약에 의한 감소 (~0-15%)
```

Vibe의 경우:
```
GPT-4o 정확도 ≈ 70% (Python/TS/Lua 전이) + 25% (프롬프트) - 5% (while/초기화 충돌)
             = 90%  (실제: 90%)
```

### 7.2 설계 결정의 비용 모델

모든 설계 결정은 아래 프레임워크로 평가해야 한다:

```
순가치 = (결정이 제공하는 가치) - (LLM 충돌 비용) - (프롬프트 토큰 비용)

여기서:
- 가치: 안전성, 일관성, 가독성 등 언어 품질 향상
- LLM 충돌 비용: 학습 데이터 대비 override 실패율 × 사용 빈도
- 프롬프트 토큰 비용: override에 필요한 추가 토큰 수 / 전체 프롬프트 예산
```

| 결정 | 가치 | LLM 충돌 비용 | 프롬프트 비용 | 순가치 |
|------|------|--------------|-------------|--------|
| `and`/`or`/`not` | 가독성 높음 | ~0% (Python 전이) | 10 토큰 | **매우 높음** |
| 들여쓰기 블록 | 간결성 높음 | ~0% (Python 전이) | 5 토큰 | **매우 높음** |
| `none` (not null) | 안전성 중간 | ~0% (Python 전이) | 5 토큰 | **높음** |
| 명시적 `return` | 명확성 중간 | ~0% (다수파 정렬) | 0 토큰 | **높음** |
| 초기화 필수 | 안전성 중간 | ~10% (다수 언어 충돌) | 30 토큰 | **중간** |
| `while` 제거 | 일관성 낮음 | ~10% (거의 모든 언어 충돌) | 40 토큰 | **낮음~음수** |

### 7.3 최종 권고

1. **즉시 조치**: `+=` 프롬프트/파서 정합성 확보, 시스템 프롬프트에 "없는 것" 섹션 추가
2. **단기 조치**: 초기화 없는 선언 대응 (프롬프트 강화 또는 문법 확장)
3. **중기 검토**: `while` 키워드 복원 여부 결정 (벤치마크 태스크 50개로 확장 후 재평가)
4. **장기 원칙**: 모든 새로운 문법 결정 시 "학습 데이터 정렬 비용" 사전 평가

**핵심 메시지**: Vibe가 학습 데이터 0으로 90%를 달성한 것은 "LLM을 위한 언어 설계"가 유효함을 증명한다. 남은 10%는 학습 데이터와의 소수 충돌 지점을 해결하면 달성 가능하다. LLM의 학습 데이터는 "중력"처럼 작용한다 -- 이에 순응하면 비용 0, 역행하면 비용이 지수적으로 증가한다. 언어 설계자는 이 중력을 적으로 삼지 말고, 활용해야 한다.
